# Oximy Gateway v0.1.0 — live test findings & fix plan (2026-06-12)

Empirical test of the shipped `v0.1.0` binary (providers: OpenRouter + Gemini, one
upstream MCP server). The governance spine is solid; the LLM data-plane is
text-only and has three concrete bugs plus one unwired capability. This doc records
root causes (file:line) and the fix plan.

## Scorecard

**Works:** LLM call, streaming SSE, cross-provider via OpenRouter (OpenAI/Anthropic/
Gemini/DeepSeek), virtual keys + **durable USD budget** (survives restart), model
allowlist (403), secrets guardrail (403), exact-match cache (MISS→HIT, $0),
telemetry (overview/usage/logs), MCP federation + namespacing + `tools/call`, MCP
admin readback.

**Broken / unwired:**
1. **Tool / function calling** — dropped, never reaches the provider.
2. **Multimodal (image parts)** — `400 "expected a string"`, rejected at the door.
3. **Native Gemini** — `400 "no egress configured for provider google"`.
4. **Per-tool MCP ACL** — engine exists, nothing populates it.

## Root causes & fixes

### 1. Tool / function calling (two layers)
- **HTTP boundary** `crates/gateway-control/src/wire.rs`: `WireChatRequest` has no
  `tools`/`tool_choice`; `WireMessage.content` is `String`; `to_unified()` builds
  `Message::text(...)` only. `WireChatResponse`/`WireOutMessage` has no `tool_calls`,
  so even a returned call is stripped from the client response.
- **Egress** `crates/gateway-llm/src/transports/openai.rs`: outbound `WireRequest`
  has no `tools`/`tool_choice`; `WireMessage.content` is `String` (no tool_calls /
  tool_call_id). (The response/stream side already parse `tool_calls`.)
- **Fix:** carry `tools`, `tool_choice`, assistant `tool_calls`, and tool-role
  `tool_call_id` through `wire.rs` (request + response) and serialize them in the
  OpenAI egress. Non-streaming first; streaming tool-delta emit is a follow-up.

### 2. Multimodal
- Same `content: String` chokepoints as #1 (control `wire.rs` + egress). The
  internal `ContentPart::Image{ Url | Base64 }` already exists end-to-end.
- **Fix:** accept string-or-array `content` in `wire.rs`, map array parts to
  `ContentPart`, and serialize OpenAI `image_url` parts in the egress. (Falls out of
  the #1 change once `content` becomes structured.)

### 3. Native Gemini
- `crates/oximy-gateway/src/main.rs` registers `GEMINI_API_KEY` as provider id
  **`gemini`**, but the bundled models.dev catalog keys Google models as
  **`google`**, so `registry.get(model).provider == "google"` has no egress.
- **Fix:** register the native Google transport under id `google` (the catalog id),
  and strip a leading `google/` from the model name in the Gemini egress so the
  Gemini API receives the bare model id. Keep an OpenRouter route as fallback.

### 4. Per-tool MCP ACL
- `crates/gateway-mcp/src/acl.rs` `ToolAcl` + `federation` enforcement exist and are
  unit-tested, but nothing calls `acl.set(key_id, ...)` outside tests. No admin/CLI/
  config surface, so every key calls every tool.
- **Fix:** add `tool_allowlist: Option<Vec<String>>` to the admin key-create payload
  (`admin.rs`), persist it on the key, and seed `federation.acl_mut().set(...)` on
  key create and on startup. Optional: `keys create --tool-allowlist` CLI flag.
  Result: a key allowlisted to read-only Linear tools is blocked from
  `linear__delete_issue` (the headline scenario).

## Non-blocking observations
- `x-overhead-duration-ms` ≈ total upstream latency, not gateway-only overhead —
  undercuts the "sub-1ms self-overhead" benchmark claim. Should measure only the
  gateway's own time.
- Config file honors only `routes` + `model_overrides`; `keys`/`providers`/
  `guardrails`/`mcp_servers` are silently ignored (`main.rs` `FileConfig`).
- MCP tool calls are audited to a spine sink but there's no read endpoint / dashboard
  surface.
- Telemetry counters (request totals, cost rollups) are in-memory and reset on
  restart; only keys + the budget ledger are durable.
- Docs over-promise: dashboard "Add server"/"Add provider", `oximy-gateway mcp`/
  `config` CLI, and `--config` flag don't exist in v0.1.0.
