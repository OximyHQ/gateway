# Kong AI Gateway — Competitive Intelligence Report

Date: 2026-06-10. Researched for a team building a new open-source AI gateway (unified LLM gateway + MCP gateway, single binary, dashboard, agent-first CLI/MCP control plane).

## 1. What it is

Kong AI Gateway is not a standalone product — it is a **plugin layer on top of Kong Gateway** (the OpenResty/Nginx-based API gateway, written in Lua, Apache 2.0, ~43.6k GitHub stars / 5.1k forks at Kong/kong). Kong positions it as covering "the full AI data path" with three traffic patterns under one umbrella:

- **LLM Gateway** — app-to-model traffic (ai-proxy / ai-proxy-advanced + ~23 AI plugins)
- **MCP Gateway** — agent-to-tool traffic (AI MCP Proxy, AI MCP OAuth2, autogenerated MCP servers from REST APIs; introduced in Gateway 3.12)
- **Agent Gateway** — agent-to-agent (A2A protocol) traffic; GA in AI Gateway 3.14 (April 14, 2026)

Marketing claim: "60+ AI features." Control plane is Kong Konnect (SaaS) with an "AI Manager" UI; data plane can run self-hosted/hybrid/Dedicated Cloud Gateways.

## 2. Full feature surface (by area)

### 2.1 LLM proxying & routing
- **AI Proxy** (OSS): unified API to many providers; transforms requests/responses so apps can switch models without code changes. Route types: llm/v1/chat, completions, embeddings, plus newer ones (audio, image, batch, assistants, files, rerank).
- **AI Proxy Advanced** (Enterprise): multi-target **load balancing** with 6+ algorithms — round-robin, consistent hashing, lowest-latency, usage-based (token-aware), weighted, and **semantic routing** (embed the prompt, route to the model whose description/examples are most similar).
- **Dynamic model routing via `model_alias`** (3.14): clients ask for "cheap"/"powerful" names; Kong maps to concrete provider models — decouples client naming from provider IDs.
- **Retries/failover/fallbacks** across targets; canary releases between models.
- **Streaming**: token-by-token SSE passthrough with reduced TTFT (expanded in 3.11).

### 2.2 Providers (AI Proxy Advanced list, mid-2026)
OpenAI, Azure AI/OpenAI, Anthropic, AWS Bedrock (incl. Converse, RetrieveAndGenerate), GCP Vertex AI, Gemini, Cohere, Mistral, Hugging Face, Llama (self-hosted), Ollama, vLLM, Databricks, DeepSeek, xAI, Alibaba DashScope, Cerebras, plus generic OpenAI-compatible endpoints. Embeddings across Cohere/Mistral/Gemini/OpenAI-compatible.

### 2.3 Multimodal & extended LLM APIs (3.11, mid-2025)
- Audio: speech-to-text, realtime translation, text-to-speech (Azure, OpenAI, Gemini, Bedrock).
- Vision: image generation + mask-based image editing (Azure, OpenAI).
- **Batch API**, **Assistants API**, **Files API** proxying (parallel LLM calls, multistep workflow orchestration, persistent document storage).
- **Rerank API** for RAG relevance.
- Response metadata exposure (logprobs, function-call payloads) for observability.

### 2.4 Prompt engineering & cost reduction
- **AI Prompt Template** (OSS): fill-in-the-blank parameterized prompts.
- **AI Prompt Decorator** (OSS): prepend/append chat messages (system prompt injection at gateway).
- **AI Prompt Compressor** (Enterprise, 3.11): compresses prompts (LLMLingua-style) — claims up to **5x cost reduction retaining ~80% semantic fidelity**; aimed at system prompts and RAG context.
- **AI Semantic Cache** (Enterprise): embeds prompts, stores in vector DB (Redis, Valkey Vector Search added 3.14), serves semantically-similar cached responses.
- **AI RAG Injector** (Enterprise): gateway-side RAG — auto-inject vector-DB content into prompts so apps don't implement RAG.

### 2.5 Guardrails / security / data governance
- **AI Prompt Guard** (OSS): regex allow/deny lists on prompts.
- **AI Semantic Prompt Guard** (Enterprise): embedding-similarity topic allow/deny lists.
- **AI Semantic Response Guard** (Enterprise): same on responses.
- **AI PII Sanitizer** (Enterprise): detect/redact PII in requests and responses (can re-insert on the way back).
- Third-party guardrail plugins: **Azure Content Safety, AWS Bedrock Guardrails, GCP Model Armor, Lakera Guard**, and **AI Custom Guardrail** (config-driven integration to any guardrail API, e.g. NVIDIA NeMo) — standardized guardrail semantics across providers in 3.14.
- **AI LLM-as-Judge** plugin (response quality scoring).
- All ~100+ classic Kong plugins also apply: OIDC, mTLS, JWT, ACL, IP restriction, bot detection, etc.

### 2.6 Rate limiting & cost control
- **AI Rate Limiting Advanced** (Enterprise): token-based (not just request-based) limits per provider/model/consumer; 3.14 added **global limits aggregated across all providers + per-model caps** (e.g. "1M tokens/day total, max 200K to GPT-4o").
- Konnect: cost dashboards, budget alerts, showback/chargeback positioning (FinOps solution pages).

### 2.7 MCP Gateway (3.12+)
- **AI MCP Proxy**: protocol bridge MCP↔HTTP — exposes existing REST APIs as remote MCP servers **without writing server code** (autogeneration), and gateways third-party MCP servers; tools defined from Kong routes/OpenAPI.
- **AI MCP OAuth2**: OAuth 2.1 per the June-2025 MCP spec — Kong acts as the OAuth Resource Server, serves protected-resource metadata to MCP clients; JWK-based local token validation (3.14) for auth servers without introspection; **RFC 8693 token exchange** for downscoping agent tokens; **scope-based tool filtering** (restrict which MCP tools a token can see/call via OAuth scopes).
- MCP observability: per-tool usage tracking, prompt/completion sizes (context-bloat detection), server latency/error/throughput; `ai_mcp_reqs` metric.
- Roadmap items announced: semantic tool selection, domain-specific tool bundles, bundle-level policy.

### 2.8 Agent Gateway / A2A (3.14, GA Apr 2026)
- **AI A2A Proxy** plugin: governs Google A2A-protocol agent-to-agent traffic — agent identity verification/authn, real-time message inspection (policy violations, prompt-injection detection), per-agent token cost attribution, audit logging, unified observability across LLM + MCP + A2A.

### 2.9 Observability
- L7 AI metrics: tokens per provider/model/consumer, latency, error rates, **cost**; OpenTelemetry span attributes + metrics; structured AI audit logs (payload logging).
- **Konnect Advanced Analytics / AI Manager**: pre-built LLM dashboards (LLM latency, costs, prompt/completion tokens; filter/group by provider, request model, response model, team), custom dashboards, tool-call analytics, **agentic traffic map** (visualize which agents consume which LLMs/MCP servers), LLM catalog curation with access tiers.

### 2.10 Config, deployment, control plane
- Deployment: self-hosted (Postgres-backed or **DB-less declarative YAML**), hybrid CP/DP, Konnect SaaS control plane, Dedicated Cloud Gateways, Kubernetes via Kong Ingress Controller (CRDs); single docker-compose quickstart; ports 8000 (proxy) / 8001 (Admin API) / 8002 (Kong Manager UI).
- Config surfaces: **Admin API** (REST), **decK** (declarative CLI, gitops), **Terraform provider**, Konnect Control Plane Config API, Kong Manager / Konnect AI Manager GUIs, KIC CRDs. Secrets via vaults (env, HashiCorp, AWS/GCP/Azure, Konnect Config Store).
- Extensibility: plugin SDK in **Lua** (native), plus external plugin runners for Go/Python/JS (slower, less used). No WASM-first story for AI plugins.

### 2.11 Agent-facing control plane (AX)
- **Kong Konnect MCP Server** (Kong/mcp-konnect, open source, TypeScript): lets Claude/agents manage Konnect by natural language. Notable design: only **3 meta-tools — `search` (find relevant API operation), `get_schema`, `execute`** — dynamically exposing the whole Konnect API instead of hundreds of static tools. Auth via PAT or service-account token. Marketed use cases: auto-configure plugins/policies, map services, debug traffic, query analytics.
- Plus the data-plane MCP features above (auto-generate MCP servers from APIs) — Kong is explicitly courting agents as both *consumers* (MCP/A2A gateway) and *operators* (Konnect MCP server).

## 3. Licensing & pricing
- **OSS (Apache 2.0, Kong/kong)**: core gateway + ~6 free AI plugins (ai-proxy, ai-request-transformer, ai-response-transformer, ai-prompt-guard, ai-prompt-template, ai-prompt-decorator) — shipped in Gateway 3.6 (Feb 2024).
- **Enterprise-only**: ai-proxy-advanced (load balancing/semantic routing), ai-semantic-cache, ai-semantic-prompt/response-guard, ai-rate-limiting-advanced, ai-pii-sanitizer, ai-rag-injector, ai-prompt-compressor, MCP plugins, A2A plugin, all the Konnect analytics/AI Manager. The MCP Gateway is explicitly "enterprise-only … utilizing paid plugins."
- **Konnect pricing**: free tier (small PoC limits); Plus ≈ $105/mo per Gateway Service + ~$200 per extra 1M requests; Enterprise = custom (commonly $50k+/yr reported). Gotcha reported by third parties: **each LLM model routed to can count as a separate billable "service."**

## 4. Performance (published)
- Kong's own benchmark (Gateway 3.10, EKS c5.4xlarge, WireMock backend, K6, 400 VUs, 1000-token prompts): Kong ~26,000 RPS, P95 ~8ms, P99 ~12ms vs Portkey ~11,000 RPS / P95 ~23ms and LiteLLM ~3,000 RPS / P95 ~60ms — "228% faster than Portkey, 859% faster than LiteLLM," 65–86% lower latency, same 12-CPU allocation. (Vendor benchmark; proxy overhead only, mocked LLM.)
- Prompt Compressor: up to 5x token-cost reduction at ~80% fidelity (vendor claim).

## 5. Weaknesses / complaints (third-party + community)
1. **OSS→Enterprise paywall cliff**: teams adopt OSS then hit a five-figure contract for one need (OIDC, semantic cache, token rate limiting, MCP gateway). Most of the differentiated AI surface is paid.
2. **Operational heft**: full deployment wants Postgres or a nontrivial K8s controller setup; analysts estimate 0.25–0.5 FTE just to run Kong-on-K8s. "Nginx-level throughput you don't need when the LLM is the bottleneck."
3. **Lua extensibility tax**: custom guardrails/policies require writing Lua Kong plugins (or slower external plugin runners) — repeatedly cited as a hiring/skills problem vs Python-native LiteLLM.
4. **Pricing opacity + per-service billing**: hidden costs, per-Gateway-Service metering where each model endpoint may bill separately; enterprise pricing is contact-sales only.
5. **Config sprawl**: AI behavior is assembled from 23 separate plugins, each with its own config schema, ordering and interaction rules — high cognitive load vs a purpose-built gateway with one coherent config.
6. **Konnect dependency for the good UX**: AI Manager, LLM analytics dashboards, agentic map, budget alerts are SaaS-control-plane features; pure self-host gets Admin API + basic Kong Manager.
7. **API-gateway heritage**: AI is a plugin retrofit; some LLM-native concerns (e.g. provider-aware retry semantics, response-shape drift across providers, eval workflows) are thinner than in LLM-first gateways; no prompt-management/eval/playground story.
8. **Benchmark caveats**: their headline perf numbers are self-published against a mocked backend, measuring proxy overhead — least relevant dimension when upstream LLM latency is seconds.

## 6. Notable release timeline
- 3.6 (Feb 2024): AI Gateway launch, 6 OSS AI plugins.
- 3.8 (2024): semantic caching, semantic prompt guard, semantic routing, 6 LLM LB algorithms, Bedrock/Vertex.
- 3.10 (2025): PII sanitizer, RAG injector improvements, benchmark release.
- 3.11 (mid-2025): prompt compressor, multimodal (audio/image), Batch/Assistants/Files, Hugging Face, AWS guardrails, AI Manager UI.
- 3.12 (late 2025): MCP Gateway (AI MCP Proxy + OAuth2, MCP observability, MCP server autogeneration from REST APIs).
- 3.14 (Apr 2026): Agent Gateway GA (A2A), token exchange RFC 8693, scope-based MCP tool filtering, JWK validation, model_alias dynamic routing, global+per-model token rate limits, standardized guardrails, Databricks/DeepSeek/vLLM/Valkey.

## 7. Lessons for our build
- **Steal**: semantic routing/caching as first-class primitives; token-aware (not request-aware) rate limiting with global+per-model budgets; MCP-server autogeneration from OpenAPI; OAuth2.1 resource-server posture + scope-based tool filtering + token downscoping for agents; the 3-meta-tool (search/get_schema/execute) MCP control-plane pattern; agentic traffic map; model_alias indirection; per-agent cost attribution.
- **Exploit**: their paywall (ship semantic cache/guardrails/token limits OSS), Lua plugin friction (offer Python/TS/WASM extensibility), 23-plugin config sprawl (one coherent config), Konnect-only dashboard (ship the dashboard in the single binary), and DB requirement (single binary + embedded state).
